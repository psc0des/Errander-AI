How a maintenance batch works — end to end
①

Schedule fires

APScheduler triggers a maintenance batch on the configured cadence. The agent first checks maintenance windows — it refuses to run outside the allowed time range, and requires --force with a mandatory reason to override.

Configurable: nightly, weekly, or on-demand via CLI
›
②

AI analyzes & proposes

Layer A (LLM-powered) reads Prometheus metrics, ELK logs, and audit history across all target VMs. It produces a prioritized maintenance plan with per-VM action recommendations. Falls back to deterministic rules if the LLM is unavailable — the agent is never blocked.

AI proposes only. Never executes anything.
›
③

Human approves

The operator receives a Slack message (or Web UI alert) listing exact objects to be changed — image SHA hashes, package names with version, file paths. Reacts ✅ to approve or ❌ to reject. The plan snapshot is hashed and stored before execution begins.

30-min timeout → auto-REJECT. Bulk category approvals are rejected.
›
④

Deterministic execution

Layer B re-validates the plan hash (drift detection between approval and execution time), acquires a VM-level lock, then executes only the approved exact objects via SSH. No LLM, no AI-generated shell commands in the execution path. Any state drift since approval aborts that object's action.

Plan hash mismatch = immediate abort. No silent proceeding.
›
⑤

Audit & rollback

Every changed object gets its own immutable audit record with timestamp, actor, before/after state, and outcome — never one record per batch. On failure, Layer B triggers rollback per action policy (patching: full package-list restore). Critical alert fired if rollback itself fails.

How the approval protocol works

Every live infrastructure change — regardless of risk tier — must complete this sequence. The protocol is enforced in code, not policy.

Step 1

Plan generated
Layer A assembles a plan listing exact objects: image SHA256s, package names + versions, file paths. Plan is hashed and snapshot saved to Audit DB.

→
Step 2

Operator notified
Slack message posted to #errander-approvals with full object list and signed Web UI link. Operator sees exactly what will change.

→
Step 3

React ✅ or ❌
Operator reacts in Slack or clicks approve/reject in Web UI. Agent polls every 30 seconds. 30-minute silence → auto-REJECT. No action taken.

→
Step 4

Drift re-check
At execution time, Layer B re-validates the plan hash and re-queries each approved object's current state. Any drift since approval → abort that object's action, not silent proceed.

→
Step 5

Execute + audit
Only approved exact objects are acted on. Each removed or changed object gets its own audit record. Rollback fires on first failure.

↙
Reject / timeout

No action taken
Rejection or timeout is recorded in Audit DB. No changes made to any VM. Operator can re-trigger on next schedule.

Safety guarantees — engineered into the platform, not configurable away

Exact-object approval

Approval artifacts must name exact object IDs (image SHA, package name + version, file path) — not action categories. The platform rejects vague bulk approvals at the protocol level.

Double drift-gate

Snapshot hash checked between approval and execution (plan-level). Each approved object re-queried on the VM at execution time (object-level). Any mismatch aborts that object — never proceeds silently.

VM-level locking

Each target VM is locked before any execution. Concurrent batches cannot race. Lock is always released on completion, failure, or timeout — never leaked.

No LLM in execution path

Layer B is pure deterministic Python. No LLM calls, no AI-generated commands, no MCP tool calls. LLM outage cannot affect in-flight execution.

Full rollback on failure

Patching snapshots the full package manifest before touching anything. Docker re-pull path defined. On failure, agent rolls back. Critical alert fired if rollback itself fails.

Per-object audit trail

Every changed or attempted object gets its own immutable audit record. Batch-level records do not substitute. Compliance-ready — records include before/after state and execution actor.

Network topology — no public attack surface

The agent VM has no public IP. Every outbound connection is HTTPS to Slack only. SSH to target VMs is within the VPN. No inbound traffic is ever expected or accepted.

VPN — Private Network

  • Controller VM (agent host) — no public IP
  • Target VMs (web-01, db-01, app-01, …)
  • Self-hosted vLLM endpoint (optional GPU VM)
  • Prometheus + Grafana monitoring VM (optional)
  • ELK / Loki log aggregation (optional)

Public Internet — outbound only

  • Slack API — approval polling (outbound HTTPS)
  • Cloud LLM API — optional alternative to vLLM
  • LangSmith — optional Layer A trace egress

SSH — Only Mutation Path

  • Key-based auth only · never passwords
  • SSH is the single channel Layer B uses to change a VM
  • All commands are pre-approved templates — no LLM-generated shell
  • Connection pool managed by asyncssh
